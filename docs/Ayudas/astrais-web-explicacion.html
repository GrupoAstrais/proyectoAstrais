Proyecto Astrais / Cliente web

Explicacion completa de astrais-web

Este documento describe, de forma profunda y ordenada, todo lo que contiene astrais-web: arquitectura, rutas, autenticacion, API, modelos, pantallas, componentes, estado local, estilos, minijuegos, logros, tienda, perfil, build, despliegue y puntos pendientes detectados al leer el codigo fuente.

React 19 TypeScript Vite Tailwind CSS 4 Axios React Router

La web es una SPA gamificada: tareas, grupos, tienda, perfil, logros y arcade comparten identidad visual y datos del backend.

01 / Vision global

Resumen Ejecutivo

astrais-web es el cliente web de Astrais, una aplicacion de productividad gamificada. Su objetivo es convertir tareas, habitos y objetivos en una experiencia visual de progresion: el usuario inicia sesion, gestiona tareas personales, colabora en grupos, recibe XP y ludiones, compra cosmeticos, personaliza temas, juega minijuegos y consulta logros.

Entrada src/main.tsx
Router src/App.tsx
API base :5684
Build tsc -b + vite
Idea central: la web no es una pagina estatica. Es una SPA protegida por JWT que consume el backend Ktor de Astrais, mantiene parte del progreso ludico en localStorage y aplica temas visuales dinamicos recibidos desde la tienda.

Que cubre realmente el proyecto

  • Autenticacion: login por email/password, registro con confirmacion, OAuth Google y refresh token automatico.
  • Dashboard: vista de inicio con acciones rapidas, tareas pendientes, tienda, notificaciones, logros y acceso a minijuegos.
  • Tareas: tareas unicas, habitos, objetivos compuestos, filtros temporales, calendario, completar/descompletar y recompensa visual.
  • Grupos: listado, creacion, union por codigo/enlace, miembros, roles, invitaciones, auditoria y tareas grupales.
  • Tienda: catalogo obtenido del backend, compra, equipamiento y aplicacion de temas.
  • Minijuegos: visor por iframe, comunicacion por postMessage, estadisticas locales y tres cabinas implementadas.
  • Logros: catalogo local calculado a partir de estadisticas del arcade, filtros y recompensas reclamables en local.
  • Perfil: datos reales de usuario combinados con estadisticas mock, edicion de nombre, preferencias, compartir, amigos y cierre de sesion.

02 / Arquitectura

Arquitectura De La Aplicacion

La aplicacion esta organizada como una SPA de React. Vite sirve el HTML base, React monta el arbol de componentes, React Router decide que pantalla se muestra y una capa centralizada de Axios encapsula la comunicacion con el backend.

1

index.html define #root, carga fuentes de Google y apunta a /src/main.tsx.

2

main.tsx crea la raiz de React, activa StrictMode y envuelve la app con BrowserRouter.

3

App.tsx declara rutas, guards de autenticacion, redirecciones y carga el tema actual al cambiar de ruta.

4

Las paginas consumen funciones de src/data/Api.ts, componentes comunes y tipos de src/types.

5

El estado persistente se reparte entre backend, tokens JWT y varias claves de localStorage para recompensas, logros y estadisticas del arcade.

Capas principales

Presentacion

Paginas y componentes TSX. Usa Tailwind utility classes, assets PNG/JPG/SVG y variables CSS globales.

Aplicacion

Logica de estado con hooks, guards de ruta, filtros, construccion de requests y sincronizacion local.

Integracion

Api.ts centraliza Axios, tokens, refresh, endpoints y utilidades de tareas.

Decisiones arquitectonicas visibles

  • Las rutas protegidas dependen de localStorage.jwtToken, no de un estado global React.
  • El tema visual se aplica escribiendo variables CSS en document.documentElement.
  • Las tareas usan optimismo parcial: tras crear/editar/completar, se actualiza el array local sin recargar toda la pagina.
  • Los minijuegos viven dentro de la propia SPA, pero se cargan en iframe para aislar la cabina del panel de arcade.
  • Logros y estadisticas de minijuegos son locales; no se sincronizan aun con la API de usuario.

06 / Integracion backend

Capa API

src/data/Api.ts cumple dos papeles: cliente HTTP y libreria de utilidades de negocio. Esta mezcla es importante: ahi conviven endpoints reales, conversiones de formularios, calculos de XP, filtros temporales, normalizacion de fechas y operaciones locales sobre arrays de tareas.

Configuracion de Axios

  • API_BASE_URL = http://${window.location.hostname}:5684.
  • instance envia JSON y tiene timeout de 10 segundos.
  • Interceptor de request: añade Authorization: Bearer jwtToken si hay token.
  • Interceptor de response: ante 401 intenta /auth/regenAccess usando jwtRefreshToken.
  • Cuando hay varios 401 simultaneos, usa cola failedQueue para esperar el refresh en curso.
  • Si el refresh falla, borra tokens y redirige a /login.

07 / Datos

Modelo De Dominio En El Frontend

UserData

Contiene id, nombre, nivel, xpActual, xpTotal, ludiones, personalGid, mascota equipada, tema y flag admin. Es la fuente principal para dashboard, tareas, tienda, tema y perfil.

ITarea

Modelo central de tareas. Incluye id, gid, titulo, descripcion, tipo, prioridad, extras por tipo, estado, recompensas y fechas de creacion/actualizacion/completado.

IGroup

Representa un grupo renderizable: gid, nombre, descripcion, foto, miembros, tareas y rol del usuario actual.

StoreItemResponse

Item de tienda proveniente del backend: id, nombre, descripcion, tipo, precio, asset, tema, coleccion, rareza, comprado y equipado.

Tipos de tarea

Tipo Uso Campos extra Comportamiento
UNICO Tarea puntual o diaria con fecha limite. extraUnico.fechaLimite Disponible desde su fecha de creacion hasta la fecha limite.
HABITO Rutina repetible. extraHabito.numeroFrecuencia y frequency Puede ser diaria, semanal, mensual, horaria o anual en utilidades, aunque el modal muestra diaria/semanal/mensual.
OBJETIVO Objetivo padre o tarea compuesta. idObjetivo en tareas hijas La UI lo trata como agrupador de subtareas.

Prioridad y recompensa

  • 0 = Easy = 20 XP.
  • 1 = Medium = 35 XP.
  • 2 = Hard = 50 XP.
  • Los ludiones vienen del backend en la tarea, aunque muchas tareas locales se crean con recompensaLudion: 0.

08 / Seguridad de sesion

Autenticacion Y Sesion

Login clasico

Login.tsx recoge email y password, llama a performLogin y guarda jwtToken y jwtRefreshToken. Si funciona, navega a /home.

Registro

Register.tsx exige contraseña de 12 caracteres con mayuscula, minuscula y numero. Primero llama a /auth/register, despues muestra campo de codigo y llama a /auth/verify.

Google OAuth

loginWithGoogle redirige al backend. El callback acepta parametros uid, hadToRegister, jwtAccessToken y jwtRefreshToken.

Refresh automatico

Si una llamada falla con 401, la app intenta regenerar access token con /auth/regenAccess. Si falla, borra tokens y vuelve al login.

Nota real del codigo: hay varios console.log que imprimen tokens o informacion sensible durante login/OAuth. Para produccion conviene retirarlos.

10 / Productividad

Sistema De Tareas

Las tareas son el nucleo funcional. La misma base se reutiliza en dashboard, pagina personal y grupos. La diferencia esta en el gid: en tareas personales se usa userData.personalGid; en grupos se usa el grupo activo.

Flujo de carga

  1. La pantalla llama a getUserData().
  2. Extrae personalGid si es vista personal, o usa activeGroup si es grupal.
  3. Llama a getTasksFromGroup(gid).
  4. Guarda las tareas en estado React.
  5. Calcula listas derivadas con getDailyTasks, getHabitTasks, filterTasksByTime y filterTasksByCompleted.

Crear y editar

El modal trabaja con ITaskFormData. Al crear, buildCreateTaskRequest transforma la forma UI al contrato del backend. Tras recibir el id, createLocalTask inserta una version local para que la UI responda inmediatamente.

Completar y recompensar

  • Al completar una tarea simple se llama a completeTask(tid) y se actualiza el array local.
  • Al descompletar se llama a uncompleteTask(tid).
  • En una tarea con subtareas, completar el padre completa primero las hijas y luego el padre.
  • Completar una subtask puede completar el padre si todas las hermanas quedan completas.
  • La notificacion de recompensa se muestra solo si la tarea no habia sido recompensada antes.

Persistencia de recompensa local

La clave rewarded-task:{gid}:{id} evita mostrar dos veces una recompensa para la misma tarea. Tambien se considera recompensada si trae fecha_completado desde backend.

Logica de fechas: las tareas unicas vencidas se ocultan en listas por defecto. Los habitos se filtran por frecuencia semanal/mensual/diaria usando diferencia de dias y fecha base.

11 / Colaboracion

Sistema De Grupos

La pagina Groups.tsx es una de las mas complejas: une carga de grupos, miembros, tareas, permisos, invitaciones, union, salida, borrado, transferencia de ownership y auditoria.

Roles

Rol numericoEtiqueta UIPermisos usados
0MiembroPuede ver grupo y tareas. No gestiona tareas ni configuracion.
1ModeradorPuede gestionar grupo, miembros, invitaciones y tareas.
2OwnerPuede cambiar roles, ceder ownership y borrar grupo.

Flujos principales

  • Cargar grupos: getUserGroup() + getUserData(), excluyendo el grupo personal.
  • Precargar miembros: para cada grupo llama a membersGroups(gid) y normaliza miembros.
  • Seleccionar grupo: activa panel y, si no esta cargado, llama a getTasksFromGroup(activeGroup).
  • Crear grupo: CreateGroupModal produce nombre/descripcion, luego createGroup.
  • Unirse: detecta si el input parece URL o codigo y usa groupJoinByLink o groupJoinByCode.
  • Invitaciones: crea, lista, muestra estado, genera URL legacy y permite revocar.
  • Auditoria: carga eventos con eventosGroup y los traduce a textos legibles.
Comportamiento notable: al unirse a un grupo desde el modal se llama a window.location.reload(). Funciona, pero rompe la continuidad SPA y podria sustituirse por recarga de estado.

12 / Gamificacion economica

Tienda, Cosmeticos Y Temas

Shop.tsx se alimenta del backend, no del catalogo local antiguo. Transforma cada StoreItemResponse en ShopDisplayItem para poder categorizar, mostrar rarezas, calcular colores de acento y explicar el perk visual.

Categorias derivadas

  • APP_THEME se muestra como Temas.
  • PET se muestra como Mascotas.
  • Si coleccion contiene event, se muestra como Eventos.
  • El resto cae en Especiales.

Tema dinamico

src/styles/theme.ts parsea un JSON con colores primary, secondary, tertiary, background, backgroundAlt, surface, text y error. Solo acepta colores hex validos #RRGGBB. Despues escribe variables CSS --astrais-* en el root.

Flujo de compra/equipamiento

  1. Carga en paralelo getUserData() y getStoreItems().
  2. Aplica el tema del usuario o el tema default de tienda.
  3. Selecciona automaticamente el primer item visible de la pagina actual.
  4. Al comprar llama a buyStoreItem(id), refresca usuario y catalogo.
  5. Al equipar llama a equipStoreItem(id). Si el item es APP_THEME, aplica tema inmediatamente.
Catalogo local residual: shopCatalog.ts y shopStorage.ts definen una tienda local anterior con balance base 260, pero Shop.tsx ya trabaja con backend. Deben entenderse como soporte historico/prototipo salvo que se vuelvan a conectar.

13 / Arcade

Minijuegos

Los minijuegos estan implementados dentro del frontend y se integran con el catalogo mediante iframe. Cada cabina, al terminar una ronda, envia un mensaje al padre con type, gameId, score y durationSeconds.

Comunicacion iframe

{
  type: "astrais:game-round-completed",
  gameId: "atrapa-ludiones",
  score: 24,
  durationSeconds: 10
}

Games.tsx valida que event.origin === window.location.origin y que el mensaje cumpla isGameRoundCompletedMessage. Despues actualiza estadisticas con buildArcadeStatsAfterRound y guarda en astrais.games.arcade.stats.

Cabinas

JuegoIDEstado en catalogoMecanica
Atrapa Ludiones atrapa-ludiones Jugable Clicker de 10 segundos. Cada click suma 1. Premio: 12 + score * 4.
Nebula Dash nebula-dash Oculto/no disponible Carrera de 5 carriles durante 30s. Flechas/A-D, escudos, fragmentos y meteoros.
Astra Memory astra-memory Jugable Memoria de secuencias con 4 pads, secuencia inicial de 3 y maximo 3 fallos.

Estadisticas locales

  • bestScore: mejor puntuacion global o por juego.
  • lastScore: ultima puntuacion.
  • gamesPlayed: partidas jugadas.
  • totalScore: suma historica.
  • totalLudionsEarned: ludiones ganados por minijuegos.
  • Hay migracion desde clave legacy astrais.games.clicker.stats.

15 / Usuario

Perfil

Profile.tsx combina datos reales del backend con datos de demostracion. Del backend usa id, nombre, nivel y XP. En local mantiene descripcion, logros, amigos, racha, focus hours y ajustes visuales.

Funciones del perfil

  • Carga inicial: getUserData() normaliza nombre, username, nivel y XP.
  • Editar perfil: valida nombre y username, llama a editUser con uid, nombreusu, lang: null y UTC offset.
  • Ajustes: cambia visibilidad, recordatorios, resumen, solicitudes, mostrar logros y compartir estadisticas en estado local.
  • Cuenta OAuth: permite definir email/password con setEmailLogin.
  • Borrar cuenta: llama a deleteUser(), borra tokens y vuelve a login.
  • Salir: borra jwtToken y jwtRefreshToken.
  • Compartir: usa Clipboard API o Web Share API.
  • Amigos: muestra una lista mock de cuatro amigos.
Estado mixto: el perfil no representa todavia todos los datos reales del backend. Varias metricas son fijas: amigos, tareas completadas, horas de foco, logros raros y grupos activos.

17 / Visual

Sistema De Estilos

El estilo mezcla Tailwind CSS 4, variables CSS propias y assets de Astrais. La identidad visual es espacial/arcade: fondos galacticos, paneles transluctos, neones, rarezas y tipografia pixel para titulos.

Archivos de estilo

index.css

Importa Tailwind y colors.css. Es el CSS global cargado desde main.tsx.

colors.css

Define variables --astrais-*, colores semanticos y mapeo @theme de Tailwind.

post-login.css

Contiene un sistema de shell post-login, paneles, nav, cards y tonos por pagina. No esta importado en App.tsx actualmente.

Tokens principales

19 / Operacion

Build, Compilacion Y Despliegue

Desarrollo

cd astrais-web
npm install
npm run dev

Vite levanta la SPA y usa HMR.

Build

npm run build

Ejecuta TypeScript project build y despues vite build, generando dist/.

Scripts propios

  • compile.sh y compile.bat: instalan dependencias, compilan y copian dist a ../docker/compiledPage.
  • deploy.sh y deploy.bat: construyen imagen con ../docker/dockerfile.nginx, paran/eliminan contenedor my-web y levantan Nginx en puerto 8080.
Atencion: los scripts de deploy llaman a docker stop my-web y docker rm my-web sin comprobar si existe. Puede mostrar errores si el contenedor no estaba creado, aunque el flujo siga dependiendo del shell.

Mejoras recomendadas

  1. Eliminar logs de tokens y datos sensibles.
  2. Separar Api.ts en cliente HTTP, servicios por dominio y utilidades puras.
  3. Conectar logros y recompensas al backend para que los ludiones reclamados sean reales.
  4. Persistir ajustes de perfil y datos sociales en API real.
  5. Revisar componentes no usados y borrar o integrar los que sean necesarios.
  6. Importar o retirar post-login.css segun el sistema visual definitivo.
  7. Corregir encoding de textos y comentarios.
  8. Ampliar responsive real de tienda, logros y minijuegos en vez de mostrar solo aviso.
  9. Añadir tests de utilidades de tareas: fechas, filtros, completar padre/subtareas y recompensas.
