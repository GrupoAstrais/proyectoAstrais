Proyecto Astrais / Backend Ktor

Explicacion completa del backend

Este documento describe toda la parte de servidor de Astrais: arquitectura, arranque, configuracion, base de datos, autenticacion, OAuth, grupos, invitaciones, tareas, recompensas, tienda, cosmeticos, avatar, administracion, SSE, recursos estaticos, herramientas de prueba y riesgos tecnicos detectados al leer el codigo fuente.

Resumen ejecutivo

El backend es una API HTTP construida con Ktor sobre Netty. Su funcion es servir la logica principal de Astrais: usuarios, login, verificacion de correo, OAuth de Google, grupos, invitaciones, tareas gamificadas, experiencia, ludiones, tienda, inventario, cosmeticos, avatar, panel de administracion, recursos estaticos y eventos SSE.

Lenguaje Kotlin
Runtime JVM 21
Framework Ktor
Persistencia Postgres

API funcional

La API esta organizada en rutas de autenticacion, grupos, tareas, tienda, avatar, administracion y SSE. La mayoria de rutas de negocio requieren el esquema access-jwt.

Dominio gamificado

El modelo de usuario incluye nivel, XP, ludiones, rachas, idioma, zona horaria, inventario, mascota equipada, tema equipado y avatar equipado.

Estado real

La compilacion Kotlin pasa, pero hay detalles de seguridad y consistencia que conviene corregir antes de usarlo como backend de produccion.

Arquitectura

La arquitectura es sencilla y directa: Ktor recibe las peticiones, valida el token si la ruta esta protegida, delega en repositorios o DAO, y finalmente Exposed ejecuta operaciones contra PostgreSQL. Tambien existe una capa de recursos estaticos y dos buses SSE en memoria.

1

Cliente HTTP

La web, Android o el panel admin llaman a endpoints JSON. La autenticacion se envia con Authorization: Bearer <token>.

2

Ktor

Application.module() instala JSON, CORS, StatusPages, JWT/OAuth, SSE y registra todas las rutas de negocio.

3

Repositorios

AuthRepoImpl, GroupRepoImpl, TaskRepoImpl y AdminDaoImpl concentran validaciones de dominio, permisos y conversiones.

4

DAO y base de datos

DatabaseDAOImpl usa Exposed DAO/DSL y transacciones suspendidas para leer y modificar tablas PostgreSQL.

5

Respuesta

Las respuestas son JSON serializadas con kotlinx.serialization, errores normalizados con Errors, o eventos SSE para cambios asincronos.

Capas reales del proyecto. No hay un framework de inyeccion de dependencias. Las implementaciones se obtienen con funciones de fabrica como getDatabaseDaoImpl(), getAuthRepoImpl(), getGroupRepoImpl() y getTaskDaoImpl(). En tests, grupos si tiene un proveedor reemplazable.

Stack y dependencias

El build esta definido con Gradle Kotlin DSL. El servidor usa Ktor, Exposed, PostgreSQL, JWT, OAuth de Google, bcrypt, cliente HTTP Apache, correo SMTP y serializacion JSON.

Arranque y configuracion

El punto central es Application.module(). La aplicacion inicializa base de datos, autenticacion, serializacion, CORS, StatusPages, SSE, cosmeticos iniciales si la instalacion es fresca y finalmente registra rutas.

Secuencia de modulo

  1. initDatabase() lee configuracion y conecta a PostgreSQL.
  2. installAuth() instala OAuth Google y JWT access/refresh.
  3. ContentNegotiation instala JSON lenient con unknown keys ignoradas.
  4. CORS permite cualquier host, credenciales, Authorization y cualquier metodo.
  5. StatusPages traduce errores JWT, token expirado, bad request y formato numerico.
  6. installSSE() activa soporte Server-Sent Events.
  7. Si la base es nueva, loadInitialCosmetics() carga recursos iniciales.
  8. routing registra auth, admin, avatar, OAuth, grupos, tareas, tienda, SSE y estaticos.

Variables de entorno/config

  • KTOR_PORT: puerto en application.yaml.
  • JWT_ISSUER, JWT_AUDIENCE, JWT_SECRET_ACCESS, JWT_SECRET_REFRESH.
  • DB_USER, DB_PASSWORD, DB_NAME.
  • GOOGLE_CLIENT_ID, GOOGLE_SECRET_ID.
  • SMTP_EMAIL, SMTP_PASSWORD y segundo par opcional.
  • UPLOAD_DIR: carpeta externa servida por /assets.
  • INVITE_BASE_URL: base publica para invitaciones seguras.
ktor:
  application:
    modules:
      - com.astrais.ApplicationKt.module
  deployment:
    port: $KTOR_PORT

jwt:
  issuer: $JWT_ISSUER
  audience: $JWT_AUDIENCE
  accessSecret: $JWT_SECRET_ACCESS
  refreshSecret: $JWT_SECRET_REFRESH
Detalle importante. application.yaml usa KTOR_PORT, pero el main() personalizado llama a System.getenv("ktor.deployment.port"). Si se arranca por ese camino, el nombre de variable no coincide con la configuracion YAML.

Base de datos

La persistencia se define con Exposed. En arranque se ejecuta SchemaUtils.create(...) para crear las tablas si no existen. La conexion apunta a jdbc:postgresql://database:5432/<dbname>.

Usuarios

Users almacena nombre, email, hash de contrasena, confirmacion, idioma, zona horaria, rol global, nivel, XP, ludiones, rachas, ultima conexion y cosmeticos equipados.

Autenticacion

UserConfirm guarda codigos de verificacion. AuthCredentials enlaza usuarios con proveedores OAuth como GOOGLE.

Grupos

Group tiene owner y flag personal. RelGroupUser guarda miembros normales/mods. Tambien hay invitaciones y auditoria.

Tareas

Task es la tabla base. TaskUnique, TaskHabit y TaskObjective guardan datos especificos por tipo.

Cosmeticos

Cosmetic guarda nombre traducido como JSON, descripcion, tipo, precio, asset, tema, coleccion y rareza.

Inventario

Inventory enlaza usuario y cosmetico comprado con fecha de compra y un indice unico usuario-cosmetico.

Autenticacion

La autenticacion combina email/contrasena con verificacion por correo, JWT de acceso y refresh, OAuth Google web, OAuth Google Android y gestion de metodos de login. El token usa el subject como UID y un claim tokenType.

Tokens JWT

  • ACCESS: expira en 10 * 60 * 1000 ms, es decir 10 minutos.
  • REFRESH: expira en 7 dias.
  • Cada tipo usa un secreto distinto: jwt.accessSecret y jwt.refreshSecret.
  • Los verificadores validan issuer, audience, firma y tokenType.

Login basico

  • Registro valida campos, idioma de 3 letras y lista ENG/ESP/RUS.
  • La contrasena se hashea con bcrypt cost 8.
  • Se envia un codigo de 6 digitos por correo usando plantilla HTML.
  • El login no devuelve tokens si el correo no esta confirmado.

Grupos

El modulo de grupos gestiona grupos personales y compartidos, permisos de owner/mod/usuario, invitaciones directas, invitaciones seguras por codigo, miembros, salida voluntaria, transferencia de propiedad y auditoria.

Roles

Los roles expuestos como enteros son 0 usuario normal, 1 moderador y 2 owner. El owner vive en Group.owner; no siempre aparece en RelGroupUser.

Permisos

Owner y mods pueden invitar, anadir y quitar usuarios. Solo owner transfiere propiedad, cambia roles y borra grupos en el flujo normal.

Auditoria

Se registran eventos como invite_created, invite_revoked, member_joined_by_invite, member_left y member_role_changed.

Invitaciones seguras

1

Creacion

randomCode() genera 24 bytes aleatorios y los codifica en Base64 URL-safe sin padding.

2

Persistencia

El servidor guarda el codigo y tambien SHA-256 hexadecimal. La busqueda y revocacion usan el hash.

3

Consumo

Al unirse por codigo se comprueba grupo, grupo no personal, pertenencia, revocacion, expiracion, maximos usos y luego se incrementa uses_count de forma atomica.

Tareas

Las tareas son el nucleo gamificado. Hay tres tipos: tareas unicas con fecha limite, habitos con frecuencia y racha, y objetivos que pueden tener subtareas. Completar tareas otorga XP y ludiones; descompletar devuelve recompensas.

UNICO

Requiere extraUnico.fechaLimite parseado como Instant. Tiene recompensa base de 50 XP y 80 ludiones antes de multiplicadores.

HABITO

Requiere extraHabito con frecuencia. Puede completarse una vez al dia en la logica actual y actualiza racha actual/mejor.

OBJETIVO

No requiere extra. Puede recibir subtareas. Si todas las subtareas estan completadas, el objetivo padre se completa y otorga recompensa.

Calculo de recompensas

multiplicadorPrioridad = 1.0 + (prioridad * 0.5)
frecuencia:
  HOURLY  = 0.2
  DAILY   = 1.0
  WEEKLY  = 5.0
  MONTHLY = 20.0
  YEARLY  = 100.0

XP base:
  HABITO = 20
  UNICO = 50
  OBJETIVO = 400

Ludiones base:
  HABITO = 25
  UNICO = 80
  OBJETIVO = 800
Economia diaria. La recompensa de ludiones tiene un limite diario de 500 para tareas normales. Los objetivos pueden saltarse ese limite. El valor realmente otorgado se guarda en ludiones_otorgados para poder revertirlo.

Rutas de tareas

Metodo Ruta Uso
POST /tasks Crea tarea en un grupo. Requiere owner o mod.
POST /tasks/{gid} Lista tareas de un grupo. Cualquier miembro puede consultar.
PATCH /tasks/{tid}/complete Completa tarea, otorga recompensas y actualiza rachas/objetivos.
PATCH /tasks/{tid}/edit Edita campos parciales y recalcula recompensa cuando procede.
DELETE /tasks/{tid}/delete Elimina tarea si el usuario tiene permisos.
PATCH /tasks/{tid}/uncomplete Revierte completado y descuenta XP/ludiones.

Estado dinamico de habitos

Al listar tareas, un habito puede responder como COMPLETE aunque la tarea base no este marcada asi en BD: se calcula comparando ultima_vez_completada con la fecha actual.

Permisos de tareas

Crear, editar y borrar exige owner o moderador del grupo. Completar y listar exige pertenecer al grupo o ser owner.

Tienda y cosmeticos

La tienda expone cosmeticos comprables con ludiones. Hay tres tipos: mascota, tema de app y parte de avatar. El inventario registra propiedad, y el usuario guarda que pet, tema o avatar tiene equipado.

PET

Usa archivo de asset, normalmente JSON/Lottie. Precio base 1000 multiplicado por rareza si no se envia precio explicito.

APP_THEME

No necesita archivo. Guarda JSON de colores en theme. Precio base 500 por rareza.

AVATAR_PART

Usa imagen de avatar. Precio base 100 por rareza. El avatar equipado se consulta desde la ruta de avatar.

Avatar

El modulo de avatar es pequeno: solo expone el avatar equipado del usuario. La respuesta incluye ID de cosmetico, nombre, referencia de imagen y rareza.

Metodo Ruta Respuesta

SSE y tiempo real

El servidor instala Ktor SSE y mantiene dos buses en memoria con MutableSharedFlow: uno global y otro por usuario. Esto sirve para avisos ligeros sin abrir WebSockets.

GlobalBusSSE

Publica eventos globales con capacidad extra de 10. Actualmente tiene shortcut publishStoreReload() para RELOAD.STORE.

UserBusSSE

Publica eventos dirigidos a un UID. Tiene shortcuts para ADDED.TASK y SIGN.OFF.

Rutas y eventos

Ruta/evento Tipo Uso
/events/global SSE publico Escucha eventos emitidos en el bus global.
/events/user SSE protegido Escucha eventos cuyo UID destino coincide con el subject del access token.
RELOAD.STORE Global Indica que clientes deben recargar la tienda.
ADDED.TASK Usuario Payload {gid, tid}, pensado para notificar tareas nuevas.
SIGN.OFF Usuario Ordena cerrar sesion, por ejemplo tras cambiar email/login.

Errores y contratos

La respuesta de error comun es Errors(errorCode, errorText), donde errorCode es el ordinal del enum ErrorCodes. Las respuestas OK sin cuerpo de dominio suelen usar {"aknowledged": true}.

Codigo Significado Uso habitual
ERR_INVALIDTOKEN Token invalido o expirado. JWT challenge, token mal firmado, subject invalido.
ERR_MALFORMEDMESSAGE JSON o parametros mal formados. BadRequestException, tipos invalidos, parseos.
ERR_RESOURCEMISSING Recurso no encontrado. Usuario, grupo, tarea, codigo o cosmetico inexistente.
ERR_RESOURCEALREADYEXISTS Recurso duplicado. Usuario ya existe, usuario ya esta unido, cosmetico ya comprado.
ERR_BLANKVALUE Campo requerido vacio. Registro, login o verificacion con strings vacios.
ERR_BADVALUE Valor no aceptado. Idioma no soportado, rol invalido, codigo incorrecto.
ERR_FORBIDDEN Permiso insuficiente. Rutas protegidas, admin, grupo o tarea sin rol necesario.
ERR_INTERNALERROR Error interno. Excepciones SQL, OAuth externo, errores desconocidos.
ERR_RESOURCENOTMODIFIED No se pudo modificar. Ediciones fallidas o borrados no aplicados.
Contrato con typo. El mapa de exito se llama OK_MESSAGE_RESPONSE y devuelve aknowledged, con la palabra escrita sin la c de acknowledged. El frontend debe respetar el contrato actual si ya depende de el.

Estado real y riesgos

Estos puntos no son teoria: salen de leer el codigo actual. No impiden necesariamente desarrollar, pero conviene tratarlos antes de una entrega final o despliegue real.

CORS permite cualquier host con credenciales

anyHost() junto con allowCredentials = true abre demasiado la superficie. En produccion deberia limitarse a dominios concretos.

Admin inicial con password conocida

En base vacia se crea admin@astrais.com con password 1234. Debe cambiarse por secreto de entorno, bootstrap manual o deshabilitarse fuera de dev.

Creacion admin de usuarios guarda password sin hash

/admin/user/create llama a createUser con la password recibida. Ese DAO almacena el string directamente; deberia aplicar hashPassword().

Validacion JWT no comprueba que el usuario exista

validateAccessToken y validateRefreshToken validan UID numerico positivo, pero la comprobacion contra BD esta comentada.

Puerto de arranque inconsistente

YAML usa KTOR_PORT, mientras initSampleServer() lee ktor.deployment.port como variable de entorno.

OAuth web hardcodeado a localhost

El callback redirige a http://localhost:5173/oauthCallback, lo que no sirve directamente para produccion sin parametrizar frontend URL.

Vinculacion OAuth comprueba proveedor globalmente

addOauthToAccount consulta si existe el proveedor GOOGLE sin filtrar por usuario, por lo que una cuenta podria bloquear a todas.

Avatar equipado no filtra por UID

getUserEquippedAvatar(uid) selecciona id_avatar_equipado sin where uid. Puede devolver el avatar de otro usuario.

Gestion de archivos cosmeticos tiene inconsistencias

getCosmeticPath(uploadDir, filename) ignora filename y devuelve carpeta. En borrados/rollback puede intentar borrar el directorio en vez del archivo.

Validacion de extension de avatar es fragil

Algunas condiciones usan !endsWith(".png") || !endsWith(".jpg"), expresion que no puede ser falsa para una extension unica. Debe ser && o una lista de extensiones permitidas.

Logs exponen datos sensibles

La inicializacion de base de datos imprime nombre de base, usuario y password en consola. Debe eliminarse o sanearse.

StatusPages generico incompleto

Hay handlers especificos, pero los handlers genericos de excepcion estan comentados. Un error no cubierto puede devolver una respuesta menos controlada.
