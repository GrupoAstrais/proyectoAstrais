Resumen ejecutivo

La aplicacion Android es el cliente movil principal de Astrais. Es una app nativa en Kotlin con Compose, arquitectura por capas, autenticacion JWT, inicio con Google, perfil, tareas gamificadas, grupos, invitaciones, tienda, inventario, logros locales, widget y soporte offline para tareas. No es solo una interfaz: contiene reglas de sincronizacion, migracion de tareas de invitado a usuario real, cache local y control de sesion reactivo.

1 Modulo Gradle Android: :app.
26 API minima. La app compila contra SDK 35.
3 Entidades Room: tareas, grupos y acciones pendientes.
5 Tabs principales en la barra inferior: inicio, tareas, crear, grupos, tienda.
Idea central. La app esta disenada como cliente reactivo: la UI observa StateFlow y Flow, los ViewModels coordinan acciones, los repositorios aplican reglas de negocio, Ktor habla con el backend y Room mantiene una fuente local para que tareas y grupos sobrevivan a cambios de pantalla y cortes de conexion.

Arranque Android

El manifest define permisos de red, la clase Application, la actividad principal, deep links, App Links, widget y la integracion de WorkManager con Hilt.

1. Manifest

Solicita INTERNET y ACCESS_NETWORK_STATE. La app usa android:name=".AstraisApp", permite backup, soporta RTL y tiene usesCleartextTraffic="true".

2. AstraisApp

Es @HiltAndroidApp, planta Timber.DebugTree en debug, inyecta un HttpClient compartido y expone Configuration.Provider para que WorkManager use HiltWorkerFactory.

3. MainActivity

Es @AndroidEntryPoint. Aplica idioma con LocaleHelper, activa edge-to-edge, recoge deep links del intent, crea los ViewModels principales y renderiza la UI Compose.

4. Tema

AstraisandroidTheme recibe el tema equipado por el usuario si esta disponible. Si no, usa el esquema oscuro o claro por defecto.

5. Navegacion

La ruta inicial es Home si SessionManager.hasAnySession() devuelve true; si no, empieza en Login.

Arquitectura general

La app sigue un patron MVVM pragmatico. Compose observa estado, los ViewModels hacen de coordinadores de caso de uso, los repositorios deciden entre red/local/offline, y las fuentes de datos son Ktor, Room y preferencias cifradas.

Compose ScreensUI declarativa
ViewModelsStateFlow + eventos
Repositoriesreglas de dominio
Data sourcesKtor / Room / prefs
BackendAPI Ktor Astrais
Compose observa estado <- ViewModel emite estado <- Repository resuelve datos <- fuentes locales y remotas

Flujo de lectura

  • La UI se suscribe con collectAsStateWithLifecycle o collectAsState.
  • Los ViewModels exponen StateFlow con datos, carga, errores y flags de dialogos.
  • Las listas persistidas, como tareas y grupos, salen de Room como Flow.
  • El repositorio refresca desde red y actualiza Room para que la UI reciba el cambio automaticamente.

Flujo de escritura

  • Si hay conexion, el ViewModel llama al repositorio y este al API remoto.
  • Si falla una tarea, se actualiza Room de forma optimista y se encola una PendingAction.
  • WorkManager reintenta mas tarde y traduce IDs temporales negativos a IDs reales.
  • En logout se limpian datos locales para evitar fugas entre usuarios.

Navegacion

La navegacion usa Navigation Compose con rutas tipadas serializables. Hay un grafo superior para autenticacion y home, y un grafo interno dentro de HomeContainer para las tabs y pantallas secundarias.

Grafo principal

  • Route.Login: pantalla de acceso.
  • Route.Register: registro y verificacion.
  • Route.Onboarding: configuracion inicial de perfil.
  • Route.Home: contenedor principal con bottom bar.
  • Route.Profile: perfil fuera del NavHost interno.

Grafo interno de Home

  • Home, TasksTab, GroupTab, StoreTab.
  • GroupDetail(gid, role, name, description).
  • GroupSettings(gid, role, name, description).
  • Inventory y Logros.
  • El boton central de la bottom bar abre el dialogo global de crear tarea.
Control de acceso. El modo invitado puede entrar a Home y Tareas locales, pero la app bloquea Grupos y Tienda con snackbar. Si una sesion deja de estar activa, el grafo principal navega a Login y limpia la pila.

API y conexion con backend

La app habla con el backend de Astrais usando Ktor Client. La URL base esta hardcodeada en data/api/KtorClient.kt como http://172.22.238.162:5684. El cliente instala JSON, logging, timeout y autenticacion bearer.

Configuracion Ktor

  • ContentNegotiation con kotlinx.serialization.
  • Logging en nivel BODY, enviando mensajes al AppLogger.
  • HttpTimeout con 15 segundos de timeout.
  • Auth bearer cargando tokens desde SessionManager.
  • Renovacion automatica con POST /auth/regenAccess.

Serializacion

  • Hilt provee Json { ignoreUnknownKeys = true; isLenient = true }.
  • Los DTOs usan @Serializable y, cuando hace falta, @SerialName.
  • Las respuestas de error se normalizan con ServerErrorResponse.
  • Los modelos de dominio convierten DTOs y entidades Room a tipos usados por la UI.

Acoplamiento de entorno. La URL base apunta a una IP privada por HTTP. Esto sirve para desarrollo en red local, pero en produccion deberia externalizarse por flavor/build config y usar HTTPS sin usesCleartextTraffic.

Sesion, autenticacion y seguridad

La sesion se gestiona en SessionManagerImpl, que usa EncryptedSharedPreferences con MasterKey. La UI observa StateFlow para saber si hay sesion activa y si es modo invitado.

Datos guardados

  • access_token y refresh_token.
  • personal_gid del grupo personal.
  • is_guest para modo invitado.
  • pending_deep_link para invitaciones recibidas antes de login.

Modo invitado

  • No usa tokens de backend.
  • Activa isSessionActive para permitir entrar a Home.
  • Usa SessionManager.GUEST_GID = -1.
  • Permite tareas locales y bloquea grupos/tienda.

Promocion a usuario

  • Despues de login/registro, guarda tokens reales.
  • Obtiene /auth/me para recuperar datos de perfil.
  • Guarda personalGid.
  • Migra tareas de invitado al backend si era necesario.
Componente Comportamiento Observacion tecnica
Ktor Auth Incluye access token en rutas salvo login, register y regenAccess. Si falla la renovacion, limpia la sesion completa.
AuthRepository Guarda tokens, consulta perfil y decide si hace falta onboarding. Considera onboarding si el nombre contiene @, es Viajero o NUEVO_USUARIO.
SessionOrchestrator Coordina logout y post-login. En logout borra sesion, tareas locales y grupos locales.
MainViewModel Observa sesion, procesa deep links y comprueba onboarding. Tambien usa snackbar global para errores o acciones de invitacion.

Room y estrategia offline

La app no depende solo de la red. Usa Room como cache y como registro de cambios pendientes. El soporte offline se centra especialmente en tareas; grupos se cachean para listado, pero sus escrituras dependen del servidor.

Entidad Tabla Campos importantes Funcion
TareaEntity tareas ID, titulo, descripcion, tipo, estado, prioridad, recompensas, padre, datos extra, isPendingSync Representa tareas locales y remotas; IDs negativos indican creaciones temporales.
GrupoEntity grupos ID, nombre, descripcion, rol Cache reactiva de los grupos del usuario.
PendingAction pending_actions Tipo, JSON, targetId, createdAt, retryCount, lastError Cola de operaciones que deben enviarse al backend cuando vuelva la red.

TareaDao

Expone tareas como Flow, inserta y reemplaza listas, marca completado o activo, edita detalles, borra, consulta subtareas y realiza transacciones que actualizan la tarea y encolan acciones al mismo tiempo.

GrupoDao

Guarda el listado local de grupos, borra uno, limpia todos y reemplaza la lista completa cuando se refresca desde el servidor.

ActionDao

Gestiona la cola FIFO de acciones pendientes, incrementa reintentos, guarda errores, actualiza IDs temporales y elimina acciones agotadas.

Migraciones. La base astrais.db esta en version 2 y usa fallbackToDestructiveMigration(). Si cambia el esquema sin migracion formal, Room puede destruir los datos locales del usuario.

Sincronizacion offline

La sincronizacion combina actualizaciones optimistas, cola local y WorkManager. El usuario ve cambios inmediatos en tareas; si el backend no esta disponible, la accion queda pendiente y se reintenta despues.

Accion UI

El usuario crea, completa, descompleta, edita o borra una tarea.

ViewModel

TaskViewModel intenta la operacion directa o llama al metodo optimista si es invitado, offline o hay fallo.

Room

La tarea se actualiza localmente y se guarda una PendingAction con el tipo de operacion.

WorkManager

scheduleSync(context) programa SyncWorker con red requerida y backoff exponencial.

SyncWorker

Procesa acciones por orden, traduce IDs temporales, llama al backend y borra acciones exitosas.

Tipo de accion Operacion remota Detalle de consistencia
CREATE_TASK POST /tasks Usa ID local negativo, luego actualiza el ID real y referencias de subtareas.
COMPLETE_TASK PATCH /tasks/{tid}/complete Puede completar automaticamente el objetivo padre si todas sus subtareas quedan completas.
UNCOMPLETE_TASK PATCH /tasks/{tid}/uncomplete Si una subtarea se reactiva, tambien puede reactivar el objetivo padre.
EDIT_TASK PATCH /tasks/{tid}/edit Guarda un JSON con EditTareaRequest.
DELETE_TASK DELETE /tasks/{tid}/delete Un 404 se trata como exito porque el recurso ya no existe.

Clasificacion de errores

SyncWorker considera permanentes los codigos 400, 401, 403, 404 y 409. Considera reintentables 429, 5xx o errores sin codigo. El maximo de reintentos es 3.

Migracion de invitado

Al iniciar sesion desde modo invitado, TaskRepository.migrateGuestTasksToServer reempaqueta acciones con GID -1, usa el GID personal real y encola tareas huerfanas, priorizando objetivos antes que subtareas.

Estado real y riesgos tecnicos

Estos puntos no son teoricos: salen de la lectura del codigo actual. Algunos son normales en desarrollo, pero conviene resolverlos antes de entregar una version estable o publicar la app.

Build bloqueado por configuracion local

app/build.gradle.kts abre local.properties sin comprobar si existe. En este entorno el archivo falta y la compilacion falla.

Backend hardcodeado por HTTP

BASE_URL apunta a una IP privada con HTTP, y el manifest permite cleartext traffic. Esto es util en LAN, pero debil para produccion.

Logging de cuerpos HTTP

Ktor usa LogLevel.BODY. En debug puede exponer tokens, payloads, datos de usuario o respuestas sensibles en logs.

Migraciones destructivas

Room usa fallbackToDestructiveMigration(). Si cambia el esquema, la base local puede borrarse en vez de migrarse.

Deep link pendiente incompleto

MainViewModel tiene consumePendingDeepLink(), pero en el grafo visible no aparece una llamada clara despues del login. La invitacion guardada podria no procesarse automaticamente.

Widget no conectado a datos reales

AstraisWidget muestra widget_pending_tasks con valor fijo 3. No consulta Room ni el backend.

Repositorio de inventario vacio

InventoryRepository.kt solo contiene el package. La pantalla de inventario depende de otros flujos, no de un repositorio dedicado.

Cobertura de tests minima

Los tests actuales son los ejemplos de Android Studio. No cubren login, sync offline, migracion de invitado, deep links, Room ni ViewModels.

Minigames sin ruta visible

Existe MinigamesScreen.kt con WebView y bridge JS, pero Routes.kt no define una ruta para minijuegos en el grafo actual.

Release sin minificacion

isMinifyEnabled=false en release. Puede aumentar tamano y deja menos proteccion basica frente a inspeccion del APK.
